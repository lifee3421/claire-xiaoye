# 每日复盘工作台 V2｜实现规格

> 本文是实现唯一依据。除非 Claire 明确追加需求，Codex 不得自行改变页面结构、字段、自动更新规则、积分规则或交互语义。

## 0. 当前问题与改造目标

当前“每日结算”页以 Markdown 粘贴、解析和若干结算输入为主。新版本要把它改为一个直接可填写的“复盘工作台”，同时保留原有积分结算、历史记录、日记同步、阅读同步、数学/专业课进度同步和 Markdown 存档能力。

核心目标：

1. 用户不再需要在 Markdown 中填写复盘。
2. 页面按原有复盘结构直接展示为可编辑表单。
3. 滴答 Focus 与专注笔记负责自动预填学习事实；用户可修改所有字段。
4. 项目、家庭、娱乐、运动和部分杂项允许用户直接手填。
5. 22:45 自动生成当天草稿。
6. 次日发现前一天自动数据变化时，自动刷新未手改字段，不覆盖手改字段。
7. 积分结算保留在同一页面，实时预览，最终与复盘一起保存。
8. Markdown 只作为后台生成的兼容存档格式，不再作为主要输入界面。

非目标：

- 本轮不修改任何积分公式。
- 本轮不重做奖励商城、历史记录、周总结、数学进度、专业课进度、日记档案和图书馆。
- 本轮不接入华为健康真实 API；只预留健康数据接口和来源字段。
- 本轮不让模型自动编写“今日推进”或“调整”。只有真实专注笔记或用户输入可进入这些字段。

## 1. 页面命名与导航

侧边栏导航仍保留“每日结算”，避免破坏已有导航和历史习惯。

页面主标题改为：

- `今日复盘与结算`

副标题：

- `系统先整理事实，你只负责修改和补充。`

页面日期由复盘日期决定，而不是浏览器打开日期。顶部必须显示明确日期切换：

- 前一天
- 日期（只读展示 + 日期选择器入口）
- 后一天
- 回到今天

默认进入北京时间当天。

## 2. 页面总体布局

### 2.1 顶部工具栏

固定在页面最上方，包含：

- 日期与星期
- 草稿状态：`未生成 / 自动草稿 / 已修改 / 已保存 / 待校准`
- `从滴答同步`
- `检查更新`
- `保存复盘并结算`
- `导出 Markdown`
- 最后同步时间
- 最后保存时间

禁止出现：

- Markdown 大文本输入框
- “识别复盘”作为主要操作
- “复制默认 Markdown”作为主要操作

旧 Markdown 导入可保留在“更多”菜单中，仅用于兼容旧数据。

### 2.2 冻结的专注信息面板

位于顶部工具栏下方，横跨页面全宽。

必须使用 `position: sticky`，页面向下滚动时始终停留在顶部工具栏下方。它是原始数据观察区，不是编辑区。

内容：

1. 今日滴答总专注时长
2. 专注次数
3. 平均单次专注时长
4. 最长单次专注时长
5. 分类环形图
6. 全天专注分布图

明确不显示：

- 专注 Session 列表
- 每条 Focus 的逐条详情
- 自动评价用户是否“高效”

分类环形图：

- 使用 TickTick Focus 的原始分类结果。
- 中央显示总时长。
- 图例显示分类、分钟数、占比。
- 未映射项统一归入“待归类”，不能静默丢弃。

全天分布图：

- 横轴 00:00–24:00。
- 每段 Focus 按开始和结束时间绘制。
- 同一分类使用稳定颜色。
- 只表达何时专注，不做 Session 列表。

面板右上角必须显示：

- `数据来源：滴答 Focus`
- `原始数据，仅供核对；下方复盘值可手动修改。`

### 2.3 下方双栏

页面主体严格两栏：

- 左栏 58%：分类复盘
- 右栏 42%：其他信息
- 栏间距 16–20px
- 桌面最小目标视口：1366×768
- 小于 1000px 时允许变为单栏，不能横向溢出

所有模块默认展开。用户可以不填，但不能因为空值自动折叠。

## 5. 字段状态模型

所有可自动填的字段使用统一结构：

```js
{
  value: 0 | "" | false | null,
  autoValue: 0 | "" | false | null,
  source: "ticktick_focus" | "ticktick_note" | "planner" | "huawei_health" | "default" | "manual",
  sourceRevision: "",
  manuallyEdited: false,
  editedAt: "",
  updatedAt: ""
}
```

显示规则：

- 自动预填但未修改：小型来源徽标，例如“滴答自动”。
- 用户编辑后：显示“手动修改”。
- 用户编辑过的字段提供“恢复自动值”。
- 恢复自动值会清除 `manuallyEdited`，重新使用最新 `autoValue`。
- 不要给每个输入框加大块彩色边框；来源信息保持轻量。

总时长规则：

- 默认由分项合计产生。
- 若用户直接编辑总时长，则总时长成为手动覆盖值。
- 分项之后再变化时，不覆盖手动总时长，只提示差异。

文本字段规则：

- 多条专注笔记按时间顺序去重合并。
- 用户手改后，不再自动重写。
- 自动更新出现新笔记时，放入“有新自动内容”提示，用户可选择合并。

## 6. 22:45 自动草稿流程

时区固定：`Asia/Shanghai`。

每天 22:45：

1. Cyberboss 触发当天复盘草稿生成。
2. 读取当天已经结束的 TickTick Focus。
3. 读取 Focus note。
4. 读取小椰当天排程 Snapshot，仅作为辅助来源，不把计划时长当作完成时长。
5. 聚合学习、部分工作和部分杂项。
6. 生成或更新 `dailyReviewDrafts/{date}`。
7. 不创建 settlement，不改变银行积分。
8. 雪尘发送一次复盘提醒，附小椰复盘页入口。

草稿状态：

- `auto_draft`

用户打开页面后修改字段：

- `editing`

保存复盘并结算后：

- `submitted`

同一天重复点击保存：

- 不允许创建多个独立结算。
- 进入“修订已提交复盘”流程。

## 7. 次日自动更新与修订

每天次日第一次启动，或固定于上午 10:30，检查前一天数据。

### 7.1 未提交复盘

- 未手改字段直接更新。
- 手改字段不覆盖。
- 草稿状态保持 `editing` 或 `auto_draft`。

### 7.2 已提交复盘

先更新复盘的自动来源快照，再计算积分是否变化。

#### 积分不变化

- 自动更新未手改字段。
- 记录 revision log。
- 不改变银行积分。

#### 积分发生变化

禁止静默修改银行余额。

显示校准条：

- `昨日滴答数据有更新：学习 +28min，预计积分变化 +1分。`
- 按钮：`应用更新并校准积分`
- 按钮：`保留原结算`
- 按钮：`查看差异`

应用更新必须原子完成：

1. 更新该日 settlement 的结构化复盘内容。
2. 更新积分字段。
3. 对 `profile.points` 加上差额或扣除差额。
4. 记录 before / after / delta / reason / source revision。
5. 历史记录可撤回该次校准。

不能删除旧 settlement 再新建一条，避免历史顺序和日记同步重复。

## 8. 积分结算保留方式

积分结算不拆成新页面，也不挤占第三栏。

### 8.1 页面底部全宽“结算预览”

位于双栏内容之后，横跨页面全宽。

显示：

- 今日预计积分变化
- 保存后银行余额
- 真实学习时长
- 运动时长与计分强度
- 工作时长
- 入睡时间
- 娱乐时长 / 90min
- 今日类型

积分明细：

- 学习入账
- 运动入账
- 时间价值转分
- 睡眠积分
- 运动额外积分
- 工作积分
- 日型额外奖励
- 自由娱乐积分
- 复盘归档奖励

所有积分公式继续调用已有函数：

- `calculateGeneratedMinutes`
- `calculateBankPointsAdded`
- `calculateFreeEntertainmentScore`
- `calculateWorkPoints`
- `classifyDay`
- `reviewTimelinessScore`

禁止在新组件中复制一套公式。

### 8.2 底部冻结操作条

页面滚动时，底部保留一个轻量冻结条：

- `预计 +X 分`
- `保存后余额 Y 分`
- `查看积分明细`
- 主按钮：`保存复盘并结算`

不遮挡输入框；移动端退化为普通底部区块。

### 8.3 积分输入来源

- 学习：左栏学习最终总时长
- 工作：左栏工作最终总时长
- 娱乐：左栏娱乐最终总时长
- 运动：右栏运动最终总时长 + 系统计分强度
- 睡眠：右栏最终入睡时间
- 出游日：保留现有手动标记
- 复盘归档奖励：沿用现有时间规则

禁止继续保留一组与复盘表重复的“真实学习分钟 / 运动分钟 / 实际娱乐分钟”独立输入框。积分直接读取复盘最终字段。

## 10. Markdown 兼容

保存结构化复盘时，后台自动生成 Claire 原有结构的 Markdown。

用途：

- 历史导出
- 旧解析器兼容
- 日记、阅读、数学进度、专业课进度现有链路兼容

要求：

- Markdown 必须由结构化字段生成。
- 用户不需要直接编辑 Markdown。
- “导出 Markdown”只导出当前页面最终值。
- 旧 settlement 仍可使用原 parser 展示。

旧 Markdown 导入放在“更多 → 导入旧复盘”中：

- 导入后转为结构化字段。
- 用户确认后再保存。

- 以现有小椰浅色风格为基础，不引入完全不同的设计系统

颜色：

- 白底与浅灰结构
- 绿色作为主要操作与自动数据来源色
- 淡紫作为 Focus 图表和次要标签色
- 红色仅用于错误和积分扣分
- 禁止渐变、3D 和高饱和大色块

## 13. 保存事务

点击“保存复盘并结算”时必须按顺序：

1. 校验日期。
2. 聚合所有最终字段。
3. 生成结构化 review。
4. 生成兼容 Markdown。
5. 构造 settlement input。
6. 计算积分预览与最终积分。
7. 原子写入 settlement、profile points 和 dailyReviewDraft 状态。
8. 再执行日记同步、阅读同步、数学进度、专业课进度等可恢复的后续动作。
9. 更新 Agent Snapshot review status 为 submitted。

如果第 7 步失败：

- 不得显示保存成功。
- 不得更新 Snapshot 为 submitted。
- 不得只保存复盘却漏记积分。

如果日记或阅读后续同步失败：

- settlement 保持成功。
- 显示明确的局部失败与重试入口。

## 15. 实施顺序

- 运动
