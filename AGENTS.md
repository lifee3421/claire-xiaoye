# Daily 仓库开发规则

## 项目说明

Daily 是长期维护的个人效率、复盘、奖励、学习进度、日记/阅读档案和排程系统。

- 技术栈：React + Vite + JavaScript。
- 云端主路径：Firebase Auth（Google 登录）+ Firestore。
- 未配置 Firebase 时：`localStorage` demo store 作为兜底。
- 当前主要应用入口是 `src/App.jsx`；它是单体文件，修改时必须评估跨模块回归。
- 项目全貌见 `docs/PROJECT_ATLAS.md`；当前进度见 `docs/STATUS.md`。

## 开始任务前

在改代码前必须：

1. 阅读本文件、`docs/PROJECT_ATLAS.md` 和 `docs/STATUS.md`。
2. 检查 Git 状态、分支与最新 commit。
3. 审计本轮需求直接相关的真实代码、数据读写和 UI 入口。
4. 用当前代码判断实现状态；不要把聊天记录、旧 Prompt 或按钮文案当作事实。
5. 确认本轮是功能、数据、认证、安全还是文档任务，再决定验证强度。

## 范围与变更控制

- 一轮聚焦一条完整功能链，不顺手重构无关模块。
- 不因 `src/App.jsx` 较大而擅自全项目拆分；拆分必须是明确任务的一部分。
- 不删除疑似废弃代码，除非本轮明确要求且已经审计引用路径。
- 不修改无关数据结构；迁移、批量删除、认证和安全规则改动前必须先评估数据影响。
- 模板、今日草稿、历史记录和临时预览有不同生命周期，不能混用。
- 任何新功能都应明确数据主来源、写入位置、刷新恢复方式和云端/本地模式行为。

## 数据保护与同步

- 不读取、输出、提交或写入密钥、Token、真实环境变量、真实用户数据。
- 不在验证过程中写入生产用户数据；需要演示时优先使用 demo mode 或可撤回的测试数据。
- Firestore 读写必须以当前用户 UID 为边界，保持 `users/{uid}/...` 路径隔离。
- 模板应用到今天必须深拷贝并使用新实体 ID；今日修改不能反向影响模板。
- 多设备版本落后于云端时，未来必须提示“加载云端最新版本”或“坚持覆盖”；不得静默覆盖。
- 旧数据迁移必须有兼容策略、验证步骤与可回退方案。
- `entertainmentLogs` 已确认未来下线；新功能不得把它当作娱乐统计的权威来源。

## 产品语义

- `fixed event`、`locked`、`completed` 是不同概念：固定事件是时段边界；锁定是用户要求排程器不移动；完成是执行状态。
- 模板是可长期保存、反复编辑、跨日期复用的排程资产，不是某天的执行快照。
- 模板保存任务池、顺序、优先级、分类、节奏、模板时间线、固定事件、起床与上床时间；不保存完成状态、实际开始、执行历史或 Undo/Redo。
- 手工编辑过的日记必须受到同步保护；同步覆盖必须由用户明确确认。
- 数学和专业课进度的长期方向是统一“项目管理”，但当前仍是两套独立模块；不得把未来方向写成已实现功能。
- 白天即时娱乐日志已确认下线，旧数据暂留；当前结算复盘是未来娱乐统计的权威方向。

## 排程交互不变量

- 绿色预览代表完整、合法、可提交的 patch；松手必须按同一 plan 提交。
- 预览和最终提交不得使用不同算法或不同落点。
- 合法手动拖动和 resize 不重复弹确认框。
- 拖动中不得提前修改正式 PlannerState。
- 多任务变化必须原子提交。
- 排除当前任务时只使用具体 segment/block ID，不能以任务组模糊排除。
- fixed event、locked block、completed history、bedtime 是硬边界。
- 普通手动操作不得自动触发全局重新排程。
- 冲突、压缩、交换、插入必须可从 UI 预览中看出，不能静默改变其他任务。

## 验证要求

普通代码改动完成后必须：

1. 运行 `pnpm run build`。
2. 运行本轮相关的已有测试；目前若无测试，明确说明。
3. 对高交互功能做针对性浏览器人工验证，并检查控制台。
4. 验证刷新后的持久化；如受影响，还要分别检查 Firebase 与 demo mode。
5. 不把 build 通过表述为功能体验已验证。

高风险任务还需额外检查点：数据迁移、批量删除、认证、Firestore 安全规则、恢复功能和大规模状态模型重构。

## 完成与汇报

最终回复必须真实说明：

- 修改内容与未完成内容；
- 实际验证场景和 build/test 结果；
- 修改文件、commit hash、推送和部署状态；
- 已知限制或需要用户确认的地方。

禁止把部分完成写成整轮完成，禁止用“理论上可以”代替实际验证，禁止虚构文件路径、部署结果或功能状态。
